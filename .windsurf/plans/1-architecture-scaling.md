# 🏗️ План Масштабування Архітектури (Architecture Scaling)

Цей план описує технічні кроки для забезпечення стабільності та швидкодії додатку при збільшенні обсягу даних та функціоналу.

## 1. Рефакторинг управління станом (Zustand Slices)
**Проблема**: Поточний файл `useStore.ts` є монолітом, який містить усі дані (користувач, звички, завдання, фінанси, книги тощо). Це ускладнює підтримку та викликає конфлікти.
**Рішення (Slices Pattern)**: 
- Розділити store на окремі логічні частини (slices): `createHabitSlice`, `createFinanceSlice`, `createLearningSlice`.
- Об'єднати їх у головному файлі через паттерн `bound store`. Це зробить код чистішим і модульнішим.

## 2. Міграція сховища (localStorage -> IndexedDB)
**Проблема**: `localStorage` має жорсткий ліміт близько 5 МБ. При активному веденні журналу, трекінгу книг та щоденних логах цей ліміт буде швидко вичерпано.
**Рішення**:
- Замінити стандартний механізм збереження Zustand на кастомний storage engine з використанням `idb-keyval` (IndexedDB).
- IndexedDB дозволяє зберігати сотні мегабайт даних, що ідеально підходить для офлайн-додатка з багатою історією.

## 3. Розділення коду (Code Splitting & Lazy Loading)
**Проблема**: Весь додаток компілюється в один великий JavaScript файл (bundle size > 500kb).
**Рішення**:
- Застосувати `React.lazy` та `Suspense` у файлі `App.tsx`.
- Завантажувати важкі сторінки (Аналітика, Бібліотека, Фінанси) лише тоді, коли користувач на них переходить. Це кардинально зменшить час початкового завантаження (Initial Load Time).

## 4. Підготовка до Cloud Sync (Майбутнє)
**Мета**: Дозволити синхронізацію між пристроями (ПК та телефон).
**Кроки**:
- Розробити абстрактний шар API для бази даних.
- Інтегрувати Backend-as-a-Service (наприклад, Supabase або Firebase) для зберігання стану.
- Реалізувати Offline-First підхід з фоновою синхронізацією при появі інтернету.
